### Architecture (Rules 1-5)

**Rule 1 — Ports are INTERFACES, never abstract classes:**
```java
// CORRECT
public interface ConsultarXxxInputPort {
  Mono<XxxResponse> consultar(XxxCommand command);
}

// INCORRECT — NEVER use abstract class for ports
public abstract class ConsultarXxxInputPort { // NO
  public abstract Mono<XxxResponse> consultar(XxxCommand command);
}
```

**Architectural justification:**
1. Cockburn defines ports as contracts — interfaces express contracts in Java
2. Abstract classes with only abstract methods add nothing over interfaces but impose single inheritance
3. Java 21 offers `sealed interface` if implementation restriction is needed
4. Ports must NOT have behavior — they only define the contract

**NOTE: The gold standard project (wsclientes0015) and the reference project (wsclientes0024) both use interfaces in their ports. This is CORRECT and must be replicated. The project wsclientes0007 used abstract classes, which was a deviation that must NOT be repeated.**

**Rule 2 — `domain/` has ZERO imports of Spring, SOAP, JPA, WebFlux — NO EXCEPTIONS:**
```bash
# Verification: MUST return empty
grep -r "import org.springframework" src/main/java/**/domain/
grep -r "import jakarta" src/main/java/**/domain/
```
`GlobalErrorException` uses `ErrorSeverity` (domain enum) instead of `HttpStatus`. The `ErrorSeverity -> HttpStatus` mapping is done in the infrastructure layer (error resolvers).

**Rule 3 — `application/service/` NEVER imports classes from `infrastructure/`:**
The application layer only knows about ports (interfaces) and domain models. Never infrastructure DTOs, adapters, config, or mappers.

**Rule 4 — NEVER mix JPA with WebFlux in the same project:**
- If the service uses a database (Oracle/JPA) -> Spring MVC (WAS Mode)
- If it has no database and orchestrates BANCS TX -> Spring WebFlux (BUS Mode)

**Rule 5 — NEVER use `@Autowired` — always `@RequiredArgsConstructor`:**

### Header Validation — Bancs Block (Rule 9b)

**Rule 9b — Validate that `<bancs>` block exists in `<headerIn>` BEFORE calling any BANCS TX.**

When the SOAP request does not include the `<bancs>` block inside `<headerIn>`, the service MUST NOT call the Core Adapter. Instead, return HTTP 200 with the following error structure:

```xml
<error>
  <codigo>005</codigo>
  <mensaje>El bloque bancs es requerido en la cabecera</mensaje>
  <tipo>ERROR</tipo>
  <recurso>WSClientes{NNNN}/OperationName</recurso>
  <componente>WSClientes{NNNN}</componente>
  <backend/>
</error>
```

**Why:** Without the `<bancs>` block, the Core Adapter cannot build the required corporate headers (`x-app`, `x-guid`, `x-channel`, `x-medium`, `x-session`). If the request passes through without validation, the Core Adapter returns HTTP 500 with `BancsIntegrationException` — exposing internal infrastructure details in the error message.

**Implementation:**
- In the SOAP controller or service layer, check if `headerIn.getBancs() == null` BEFORE any business logic
- If null, throw a typed exception (e.g., `MissingBancsHeaderException`) or return the error directly
- This validation runs BEFORE `validateRequest()` (identification validation) — it is the FIRST check
- The error code `005` and message `El bloque bancs es requerido en la cabecera` are fixed — adapt only `recurso` and `componente` per service

**Test case to add:**
```java
@Test
void consultarContacto_deberiaRetornarError005_cuandoBancsHeaderAusente() {
  // headerIn without <bancs> block -> error codigo=005
}
```

## IMPLEMENTATION MODE DETERMINATION

Before implementing ANY service, determine the mode:

| Criterion | Mode A: BUS (WebFlux) | Mode B: WAS (MVC) |
|---|---|---|
| Source | IBM IIB / Omnichannel Bus | WebSphere Application Server |
| Has own database | NO | YES (Oracle) |
| Framework | Spring WS + Undertow (WebFlux only for BancsClient) | Spring MVC + Undertow |
| Persistence | BANCS Core Adapter via REST | HikariCP + JPA/JDBC + Oracle |
| Paradigm | Reactive (`Mono<>`, `Flux<>`) | Imperative (blocking) |

**Quick decision rule:**
- Has own database (JPA) -> **MVC**
- No database, orchestrates 3+ BANCS TX -> **WebFlux**
- No database, volume > 100 req/s -> **WebFlux**
- Uncertain case -> **MVC** (for simplicity)
- **NEVER mix JPA with WebFlux** (Rule 4)

### Lombok — Use MINIMALLY (Gold Standard Pattern)

**The gold standard wsclientes0015 uses ONLY 2 Lombok annotations:**
- `@Getter` — when needed (rare, mostly for typed exceptions)
- `@RequiredArgsConstructor` — for constructor injection of dependencies (ALWAYS instead of @Autowired)

**DO NOT use these — Java 21 has better alternatives:**

| Lombok annotation | Use instead |
|-------------------|-------------|
| `@Data`, `@Builder`, `@AllArgsConstructor`, `@NoArgsConstructor` for DTOs/models | **Java `record`** (immutable, generates accessors, equals/hashCode, toString) |
| `@Builder` on records | Records already have a canonical constructor — use `new RecordName(...)` |
| `@Slf4j` in services | Inject `ServiceLogHelper` (matches gold standard) |
| `@Setter` for utility classes | Explicit `private` constructor |
| `@NoArgsConstructor(access = AccessLevel.PRIVATE)` for utility classes | Explicit `private FooUtil() {}` constructor |
| `@Setter` for `@ConfigurationProperties` | Java `record` with `@ConfigurationProperties` (Spring 3.x supports this) |

**Examples:**

```java
// CORRECT — domain model as record
public record HeaderRequestModel(
    @Size(max = 50) String dispositivo,
    @Size(max = 5) String canal
) {}

// INCORRECT — class with Lombok
@Data @Builder @AllArgsConstructor @NoArgsConstructor
public class HeaderRequestModel { ... }

// CORRECT — BANCS DTO as record
public record Tx061404RequestDto(
    String cifCliente,
    String tipCliente,
    String tipOperacion
) {}

// INCORRECT — DTO with @Data and Builder
@Data @Builder @NoArgsConstructor @AllArgsConstructor
public class Tx061404RequestDto { ... }

// CORRECT — utility class with explicit private constructor
public final class PhoneNormalizationUtil {
    private PhoneNormalizationUtil() {}
    public static String normalize(String phone) { ... }
}

// INCORRECT — utility with Lombok
@NoArgsConstructor(access = AccessLevel.PRIVATE)
public class PhoneNormalizationUtil { ... }

// CORRECT — service with constructor injection
@Service
@RequiredArgsConstructor
public class ConsultarClienteService implements ConsultarClienteInputPort {
    private final BancsClientePort bancsPort;
    private final ServiceLogHelper log;  // Use ServiceLogHelper, not @Slf4j
}
```

**Rule of thumb:** If you're tempted to use `@Data`, `@Builder`, `@AllArgsConstructor`, `@NoArgsConstructor`, ask yourself "could this be a record?" — the answer is almost always YES.

**Determine project type from the ANALYSIS (bank's MCP rule):**
- If the WSDL has **1 operation** → `projectType: rest` — MCP generates a `@RestController` with WebFlux
- If the WSDL has **2+ operations** → `projectType: soap` — MCP generates `@Endpoint` + `WebServiceConfig` with Spring WS

**This is enforced by the MCP:** if you pass `projectType: rest` with a multi-operation WSDL, the MCP will reject it and tell you to use `soap`.

**Critical difference between REST and SOAP modes:**

| Aspect | REST (1 operation) | SOAP (2+ operations) |
|--------|-------------------|---------------------|
| Controller | `@RestController` + `@PostMapping` | `@Endpoint` + `@PayloadRoot` |
| Header injection | Automatic (WebFlux reactive context) | **MANUAL** — headers must be extracted from SOAP envelope and passed explicitly through the call chain via `HeaderRequestModel` |
| Web server | Netty (reactive) | Undertow (servlet, blocking) |
| BancsClient call | `bancsClient.call().subscribe()` (reactive) | `bancsClient.call().block()` (blocking) |
| Config | `WebServiceConfig` NOT needed | `WebServiceConfig` + `NamespacePrefixInterceptor` REQUIRED |

**Why manual header injection in SOAP mode:** Spring WS runs on a servlet container (Undertow), which is blocking. The reactive context propagation used by WebFlux to automatically inject headers does NOT work in a blocking servlet context. Therefore, the `HeaderRequestModel` must be manually extracted from the SOAP `<headerIn>` in the controller and passed as a parameter through every layer: Controller → Service → Adapter → `BancsClientHelper.toBancsHeaders()`.

#### 4.1-REST Controller (Spring WebFlux @RestController) — for 1-operation services

Copy the pattern from `wsclientes0024`. Key characteristics:

```java
@RestController
@RequiredArgsConstructor
@RequestMapping("/IntegrationBus/soap/<NombreServicio>")
public class <NombreServicio>Controller {

    private final <XxxInputPort> service;
    private final <SoapRequestMapper> soapRequestMapper;
    private final SoapResponseHelper soapResponseHelper;

    @PostMapping(
        consumes = {
            MediaType.TEXT_XML_VALUE,
            MediaType.TEXT_XML_VALUE + ";charset=utf-8",
            MediaType.APPLICATION_XML_VALUE,
            "application/soap+xml"
        },
        produces = {
            MediaType.TEXT_XML_VALUE + ";charset=utf-8",
            MediaType.APPLICATION_XML_VALUE
        }
    )
    @BpTraceable
    public Mono<SoapEnvelopeResponseDto> <operacion>(
            @RequestBody @Valid SoapEnvelopeRequestDto soapRequest) {
        // 1. Extract headerIn and bodyIn from envelope
        // 2. Validate headers via HeaderRequestValidator
        // 3. Map to domain via soapRequestMapper
        // 4. Call service (returns Mono<Result>)
        // 5. .map() to build Mono<SoapEnvelopeResponseDto> with success payload
        // 6. .onErrorResume() for BusinessValidationException, GlobalErrorException → HTTP 200 with <error>
        // 7. Let other errors bubble up to ErrorResolverHandler → SOAP Fault HTTP 500
    }
}
```

**REST pattern specifics (different from SOAP):**
- Uses `@RestController` + `@PostMapping` instead of `@Endpoint` + `@PayloadRoot`
- **Fully reactive end-to-end** — `Mono<SoapEnvelopeResponseDto>`, NEVER use `.block()`
- Manually-written `SoapEnvelopeRequestDto`, `SoapBodyRequestDto`, `SoapEnvelopeResponseDto`, `SoapBodyResponseDto` with JAXB annotations (`@XmlRootElement`, `@XmlElement`)
- `package-info.java` in `dto/` package declares namespace prefixes (NS1 for SOAP envelope, NS2 for business namespace)
- Single adapter class per port (NO `BancsClientHelper` abstract + per-TX subclasses)
- `BancsClient` injected directly with `@BancsService("ws-txNNNNNN")` qualifier
- Error handling: `ErrorResolverHandler implements ErrorWebExceptionHandler` generates SOAP Fault envelopes for uncaught exceptions (HTTP 500)
- NO `WebServiceConfig`, NO `NamespacePrefixInterceptor` — Spring WebFlux handles XML serialization via JAXB codec
- Uses Netty (reactive server) — NO Undertow
- `build.gradle` does NOT include `spring-boot-starter-web-services` or `wsdl4j`

**REST error handling — 3-tier strategy:**

1. **Tier 1 — Business errors (HTTP 200 with `<error>` block):**
   - `BusinessValidationException` → caught in `.onErrorResume()` → response with `GenericError.codigo`, `tipo="ERROR"`
   - `GlobalErrorException` → caught in `.onErrorResume()` → response with `GenericError.codigo`, `tipo="ERROR"`
   - Returns HTTP 200 always (IIB backward compatibility)

2. **Tier 2 — Unexpected errors (HTTP 500 SOAP Fault):**
   - Any other `Throwable` bubbles up via `Mono.error(throwable)`
   - Caught by `ErrorResolverHandler` (`@Component @Order(Ordered.HIGHEST_PRECEDENCE) implements ErrorWebExceptionHandler`)
   - Routed to typed resolver: `GlobalErrorExceptionResolver`, `ResponseStatusExceptionResolver`, or `UnexpectedErrorResolver`
   - Returns `SoapFaultEnvelopeDto` serialized to XML + HTTP status code

3. **Tier 3 — `ErrorResolver<T>`:** sealed abstract class with `permits {GlobalErrorExceptionResolver, ResponseStatusExceptionResolver, UnexpectedErrorResolver}`

#### 4.1-REST-STRATEGY Strategy pattern for dual-source REST services (Bancs ↔ OCP/Stratio)

**APPLIES ONLY TO REST SERVICES (gold standard `wsclientes0024`).** SOAP services use the `BancsClientHelper` pattern directly (section 4.6) and do NOT use this.

When a REST service can fetch the same data from two sources (Bancs Core via TX, and OCP/Stratio via REST), use the **Strategy pattern** to make the source switchable at runtime via `application.yml` and to support automatic failover between them.

**Configuration toggle** (`application.yml`):
```yaml
customer:
  datasource: OCP        # OCP | BANCS
  failover:
    enabled: true        # if primary fails, try the other
```

**Four strategy combinations** (all in `infrastructure/output/adapter/strategy/`):

| Strategy | Primary | Failover | Activated when |
|---|---|---|---|
| `BancsOnlyStrategy` | Bancs | — | `datasource=BANCS` + `failover.enabled=false` |
| `BancsWithOcpFailoverStrategy` | Bancs | Stratio | `datasource=BANCS` + `failover.enabled=true` |
| `OcpOnlyStrategy` | Stratio | — | `datasource=OCP` + `failover.enabled=false` |
| `OcpWithBancsFailoverStrategy` | Stratio | Bancs | `datasource=OCP` + `failover.enabled=true` |

**Strategy contract** (`application/output/port/CustomerQueryStrategyPort.java`):
```java
public interface CustomerQueryStrategyPort {
  Mono<Customer> query(CustomerRequest request, SoapRequestContext requestContext);

  // Helper: rebuilds Customer with explicit dataSource enum (used by failover branches)
  static Customer withDataSource(Customer customer, DataSourceTypeEnum dataSource) {
    return Customer.builder()
      .cif(customer.cif())
      .identificacion(customer.identificacion())
      // ... copy all fields
      .dataSource(dataSource)
      .build();
  }
}
```


**Bean factory** (`infrastructure/config/CustomerStrategyConfig.java`):
```java

  private final CustomerPropertiesConfig customerProperties;
  private final StratioCustomerPort stratioCustomerPort;
  private final BancsCustomerPort bancsCustomerPort;
  private final CustomLogLevelHandler customLogLevelHandler;

      return failoverEnabled
        ? new BancsWithOcpFailoverStrategy(bancsCustomerPort, stratioCustomerPort, customLogLevelHandler)
        : new BancsOnlyStrategy(bancsCustomerPort);
    }
    return failoverEnabled
      ? new OcpWithBancsFailoverStrategy(stratioCustomerPort, bancsCustomerPort, customLogLevelHandler)
      : new OcpOnlyStrategy(stratioCustomerPort);
  }
}
```

**Failover strategy template** (e.g., `BancsWithOcpFailoverStrategy`):
```java
@RequiredArgsConstructor
public class BancsWithOcpFailoverStrategy implements CustomerQueryStrategyPort {
  private final BancsCustomerPort bancsPort;
  private final StratioCustomerPort stratioPort;
  private final CustomLogLevelHandler customLogLevelHandler;

  @Override
  public Mono<Customer> query(CustomerRequest request, SoapRequestContext requestContext) {
    return bancsPort.getCustomerInfo(request)
      .onErrorResume(bancsError -> {
        log("Strategy Bancs→OCP: Bancs no disponible — activando FAILOVER", bancsError);
        return stratioPort.getCustomerInfo(request, null, requestContext)
          .map(c -> CustomerQueryStrategyPort.withDataSource(c, DataSourceTypeEnum.STRATIO))
          .onErrorMap(stratioError -> {
            log("Strategy Bancs→OCP: AMBOS SERVICIOS FALLARON", stratioError);
            return stratioError instanceof GlobalErrorException g
              ? g : new GlobalErrorException("999", stratioError.getMessage(), "FAILOVER");
          });
      });
  }
}
```

**Single-source strategy template** (e.g., `BancsOnlyStrategy`) — wraps any non-`GlobalErrorException` in `GlobalErrorException("999", msg, "BANCS")` so the Controller's `.onErrorResume()` always sees a typed exception.

**The Service** uses ONLY the strategy port (NOT the individual ports):
```java
@Service

  private final CustomerQueryStrategyPort customerQueryStrategy;
  // NO direct injection of BancsCustomerPort or StratioCustomerPort
}
```

**Why this design:**
- **Operational toggle** — switch source without redeploying via `application.yml` only
- **Resilience** — if the primary downstream is degraded, automatic failover keeps the SLA without manual intervention
- **Observable fallback** — each branch logs at WARN (single failure) or ERROR (both failed) with both error messages
- **Source visibility** — `dataSource` field in `Customer` lets the caller know which backend served the response (and the response success message reflects it: "OK" vs "OK-OCP")

**Test pattern for strategies** — one test class per strategy, mock both ports, verify:
1. Primary success → secondary `never()` invoked
2. Primary fails → secondary returns customer with `dataSource = STRATIO`
3. Both fail → `GlobalErrorException` with code "999" and source "FAILOVER"
4. `GlobalErrorException` from primary preserved (NOT remapped to "999")

#### 4.1 SOAP Endpoint (Spring WS)

The input adapter uses Spring Web Services (`@Endpoint`) instead of `@RestController`. This is the correct approach for SOAP services — Spring WS handles WSDL publishing, XML marshalling, and namespace routing automatically via `@PayloadRoot`.

```java
@Endpoint
@RequiredArgsConstructor
public class <NombreServicio>Controller {

  private static final String NAMESPACE_URI = "http://bpichincha.com/servicios";

  private final <XxxInputPort> service;
  private final <SoapRequestMapper> soapRequestMapper;
  private final SoapResponseHelper soapResponseHelper;

  @PayloadRoot(namespace = NAMESPACE_URI, localPart = "<OperacionSoap>")
  @ResponsePayload
  @BpTraceable
  public <Operacion>Response <operacion>(
      @RequestPayload <Operacion> request) {

    // 1. Map JAXB request to domain command/query via soapRequestMapper
    // 2. Call the input port (service)
    // 3. Build the JAXB response via soapResponseHelper
    //    - Success: populate response body fields + headerOut
    //    - Error: catch exceptions and return structured <error> block
    //      (BusinessValidationException, BancsOperationException, etc.)
  }
}
```

**Key differences from REST controllers:**
- `@Endpoint` replaces `@RestController` — Spring WS annotation for SOAP endpoints
- `@PayloadRoot(namespace, localPart)` replaces `@PostMapping` — routes by XML namespace + local element name

**CRITICAL — Element name casing (`localPart`):** the `localPart` value MUST match EXACTLY the root element name as declared in the legacy WSDL/XSD. In the IIB legacy services this is **PascalCase** (`ConsultarContactoTransaccional01`, `ConsultarDireccionesClientes01`, `CrearModificarDireccionCliente22`, etc.), **NOT** camelCase. Example:

```java
// CORRECT — matches the legacy XSD element name
@PayloadRoot(namespace = NAMESPACE_URI,
    localPart = "ConsultarContactoTransaccional01")
public ConsultarContactoTransaccional01Response
    consultarContactoTransaccional01(...) { ... }  // Java method stays camelCase (convention)
```

Do NOT apply `decapitalize()` logic anywhere — not in the controller, not in the test XML requests, not in `gradle/postProcessWsdl.groovy`. The JAXB-generated classes and the element names must preserve the exact casing from the XSD (`<xsd:element name="ConsultarContactoTransaccional01">` → `ConsultarContactoTransaccional01` everywhere in the XML wire, class names stay PascalCase per Java convention, and the Java method name is camelCase per Java convention).

Rule of thumb:
- **XML wire / `@PayloadRoot.localPart` / xpath `local-name()` in tests / JAXB class name** = PascalCase (from XSD).
- **Java method name** = camelCase (Java convention).

Reference: fix commit `bf913b9` in `tnd-msa-sp-wsclientes0007` (2026-04-14) — reverted an earlier over-simplification that decapitalized the element to match a manipulated groovy output. The correct source of truth is the XSD, not the generated Java output after post-processing.
- `@ResponsePayload` replaces `produces/consumes` — Spring WS marshals the return type to XML automatically
- `@RequestPayload` replaces `@RequestBody` — Spring WS unmarshals the SOAP body to the JAXB type
- The controller works with JAXB-generated types directly (not SOAP envelope DTOs)
- Each SOAP operation gets its own `@PayloadRoot` method in the same `@Endpoint` class

**Critical error rule:** Errors are NEVER SOAP faults. All errors — validation, BANCS, unexpected — are returned as valid SOAP responses with an `<error>` block containing tipo='ERROR'. The service always returns HTTP 200.

#### 4.6 BancsClientHelper — Abstract base + 1 subclass per TX

Location: `infrastructure/output/adapter/bancs/helper/`

The `BancsClientHelper` is an **abstract class** (NOT a port — this is infrastructure-level plumbing). Each TX gets its own concrete `@Component` subclass that injects a `BancsClient` qualified with `@BancsService("ws-txNNNNNN")`. This design enables **per-TX circuit breaker configuration** via Resilience4j.

**Abstract base class:**
```java
public abstract class BancsClientHelper {

  private final BancsClient bancsClient;
  private final CustomLogLevelHandler logLevelHandler;

  protected BancsClientHelper(BancsClient bancsClient,
      CustomLogLevelHandler logLevelHandler) {
    this.bancsClient = bancsClient;
    this.logLevelHandler = logLevelHandler;
  }

  public <T> T execute(
      String txCode, HeaderRequestModel ctx, Object body, Class<T> responseType) {
    String guid = ctx == null ? "" : ctx.guid();
    BancsResponse<T> response;
    try {
      response = doCall(txCode, ctx, body, responseType);
    } catch (BancsOperationException e) {
      throw e;
    } catch (RuntimeException e) {
      // MANDATORY — wraps WebClientResponseException and other transport errors.
      // Without this, a 502/503/timeout from Bancs bubbles up as a raw RuntimeException
      // and the Controller's generic catch returns "999 Error interno del servicio"
      // instead of propagating the real Bancs error code/message. Observed in production
      // when Bancs returned {"title":"...","errors":[{"code":"001","message":"502 BAD_GATEWAY"}]}.
      log(CustomLogLevel.ERROR,
          "[guid: {}] Exception calling Bancs TX {}: {}", guid, txCode, e.getMessage());
      throw new BancsOperationException(
          "999",
          e.getMessage() != null ? e.getMessage() : "Bancs integration exception",
          txCode);
    }
    if (response == null) {
      log(CustomLogLevel.ERROR,
          "[guid: {}] Null response from Bancs for transaction {}", guid, txCode);
      throw new BancsOperationException("999", "Empty response from Bancs", txCode);
    }
    if (!Boolean.TRUE.equals(response.success())) {
      log(CustomLogLevel.ERROR,
          "[guid: {}] Failed response from Bancs for transaction {} - code: {} message: {}",
          guid, txCode,
          response.error() == null ? "" : response.error().code(),
          response.error() == null ? "" : response.error().message());
      throw new BancsOperationException(
          response.error() == null ? "" : response.error().code(),
          response.error() == null ? "" : response.error().message(),
          response.transactionId() != null ? response.transactionId() : txCode);
    }
    return response.body();
  }

  private <T> BancsResponse<T> doCall(
      String txCode, HeaderRequestModel ctx, Object body, Class<T> responseType) {
    BancsRequest<Object> request = BancsRequest.builder()
        .transactionId(txCode)
        .header(toBancsHeaders(ctx))
        .body(body)
        .build();
    return bancsClient.call(request, responseType).block();
  }

  private BancsHeaders toBancsHeaders(HeaderRequestModel ctx) {
    return BancsHeaders.builder()
        .bancsUser(toBancsUser(ctx))
        .homologateUser(toHomologateUser(ctx))
        .build();
  }

  private BancsUser toBancsUser(HeaderRequestModel ctx) {
    BancsRequestContext bancs = ctx == null ? null : ctx.bancs();
    if (bancs == null) { return null; }
    return BancsUser.builder()
        .teller(bancs.teller())
        .terminal(bancs.terminal())
        .institution(bancs.institution())
        .branch(bancs.branch())
        .workstation(bancs.workstation())
        .channel(firstNonBlank(bancs.channel(), ctx.canal()))
        .application(firstNonBlank(bancs.application(), ctx.aplicacion()))
        .build();
  }

  private HomologateUser toHomologateUser(HeaderRequestModel ctx) {
    if (ctx == null || StringUtils.isBlank(ctx.usuario())) { return null; }
    return new HomologateUser(ctx.usuario());
  }

  private String firstNonBlank(String first, String second) {
    return StringUtils.isNotBlank(first) ? first : second;
  }

  private void log(CustomLogLevel level, String message, Object... data) {
    logLevelHandler.log(level, Thread.currentThread().getStackTrace(), message, data);
  }
}
```

**Concrete subclass per TX (one per TX code):**
```java
@Component
public class Tx061404BancsClientHelper extends BancsClientHelper {

  public Tx061404BancsClientHelper(
      @BancsService("ws-tx061404") BancsClient bancsClient,
      CustomLogLevelHandler logLevelHandler) {
    super(bancsClient, logLevelHandler);
  }
}
```

**Key design decisions:**
- `BancsClientHelper` is **abstract** (not `@Component`) — it contains shared call/error/header logic
- Each `Tx{CODE}BancsClientHelper` is `@Component` and injects `BancsClient` with `@BancsService("ws-tx{CODE}")` qualifier
- The `@BancsService` qualifier maps to a specific webclient entry in `application.yml`, which enables per-TX base URL and per-TX circuit breaker configuration
- Adapters inject the specific `Tx{CODE}BancsClientHelper` they need (not the abstract base)

**Advantage:** Per-TX circuit breaker isolation. If TX 061404 starts failing, its circuit breaker opens without affecting TX 067010. Each TX can also have different base URLs, timeouts, and connection pools.

**Naming convention — Given-When-Then (BDD):**

```java
@Test
void givenValidRequest_whenGetCustomerInfo_thenReturnsCustomer() {

@Test

  var invalidCommand = CustomerRequest.builder()

          e instanceof BusinessValidationException

      .verify();
}
```

1. **Service** (`application/service/`) — Highest priority. Cover: happy path, each validation, each error, normalizations

3. **Output Adapters** (`infrastructure/output/adapter/`) — Cover: happy path, BancsAPIClientException, null response, response with error
4. **Mappers** — Cover: full mapping, null fields, edge cases
5. **Exception handlers** — Cover: each exception type mapped correctly

### 2.1 ESQL → Java Translation Traps (CRITICAL)

When translating ESQL to Java, these translation traps cause silent bugs:

**TRAP 1: ESQL `SUBSTRING FROM x FOR n` has BOTH start AND length.**
- ESQL: `SUBSTRING(str FROM 4 FOR 9)` = 9 chars starting at position 4 (1-based)
- Java WRONG: `str.substring(3)` = ALL remaining chars from index 3
- Java CORRECT: `str.substring(3, 12)` = chars 3 to 11 (9 chars)
- **The `FOR n` is explicit length — never use unbounded `substring(start)`.**

**TRAP 2: ESQL `ELSEIF (A) OR (B)` applies the SAME body to both conditions.**
ESQL:
```esql
ELSEIF (LENGTH(x) = 13) OR (STARTSWITH(x, '001')) THEN
    SET result = '0' || SUBSTRING(x FROM 4 FOR 10);
```
Both `A` and `B` execute `'0' || SUBSTRING(...)`. Java often splits this into two `if` blocks and loses the `'0'` prepend in one branch:
```java
// WRONG — split with inconsistent body
if (x.length() == 13) return x.substring(3);           // missing '0'
if (x.startsWith("001")) return "0" + x.substring(3);  // has '0'

// CORRECT — unified body
if (x.length() == 13 || x.startsWith("001")) {
    return "0" + x.substring(3, 13);
}
```

**TRAP 3: ESQL 1-based indexing vs Java 0-based.**
- ESQL `FROM 4` = starts at character 4 (1-based) = Java index 3 (0-based)
- ESQL `FOR 9` with start 4 = chars 4-12 (1-based) = Java `substring(3, 12)`
- Always verify: `SUBSTRING(s FROM x FOR n)` = `s.substring(x-1, x-1+n)`

**TRAP 4: ESQL `LOWER(TRIM(x))` order matters.**
- ESQL `LOWER(TRIM(x))` = trim first, then lowercase (inner call runs first)
- Java equivalent: `x.trim().toLowerCase()` — NOT `x.toLowerCase().trim()`

**TRAP 5: ESQL operates on RAW values, Java tends to pre-normalize.**
- ESQL: `IF STARTSWITH(ordenante.identificacion, '000')` — operates on raw input
- Java WRONG: `.trim()` first, then `.startsWith("000")` — different behavior for `"  000123"`
- When translating, preserve the exact operation order.
